"Oslo" PhotoStore ReadMe

PhotoStore is an application that allows you to store information about your digital photographs in the Repository, and to make this information selectively available to others.

The scenario is a photographer wants to keep information about his digital artwork in the Repository, so he can sell the photos to his customers. The customers don't want to buy a cat in a bag, so they request a preview of his work - in the form of photo metadata. After they pay, all information, including the pictures, can be made available to them. For a graphical presentation of the scenario, see PhotoStore.pptx.

The application uses the "M" toolchain (mg, mgx, m compiler and mx) to make use of a custom textual DSL for photo metadata and to load the metadata into the Repository. It also uses built-in Repository security to show public and private information about photos to selected users.

Requirements

·        Microsoft "Oslo" SDK October 2008 CTP with the Repository installed

·        Microsoft .NET Framework

·        Microsoft Visual Studio 2008

·        Microsoft SQL Server 2008 with SQL Server Management Studio

Setup

Copy the PhotoStore source files to your machine. For a detailed list of files in the project and their purpose see bottom of this document.

Usage

To Run PhotoStore

1.      Make sure you have the Repository and tools (mg, mgx, "M" compiler and mx) installed on your machine.

a.      Go to C:\Program Files and make sure you have Microsoft Oslo SDK 1.0 folder.

b.      Open Control Panel -> Programs and Features and make sure you have Microsoft "Oslo" Repository installed. Next open SQL Server Management Studio and check that you have a database called Repository in the list of databases.

c.      If you do not have Microsoft "Oslo" SDK or Microsoft "Oslo" Repository installed, visit http://msdn.microsoft.com/en-us/oslo/cc748654.aspx.

2.      Open PhotoStore.sln.

3.      Build and close solution.

4.      Go to PhotoStore\bin\Debug.

a.      You will notice that most of the files from the list of files (at the end of this document) have been copied to the output directory.

5.      Run CreateSchema.bat.

a.      This will compile Photo.m using the "M" compiler, create image called Photo.mx and install that image in the Repository using mx.exe. The schema where the photos can be stored will be created. To check that the schema is there, in SQL Server Management Studio navigate to the Repository, and in the tables list verify you see the following:

                                          i.     Gallery.Photographers

                                         ii.     Gallery.Photos

                                        iii.     Gallery.FullSizePhotos

                                        iv.     Gallery.Photos_Authors

6.      Run PhotoStore.exe.

7.      Drag and drop photos into current directory and observe the application.

How to use PhotoStore and what happens when you do

When you run the application, a console window appears. It informs you that it's compiling grammar.

---

Compiling grammar...

<command line for grammar compilation>

 

Grammar OK

---

In this step, Gallery.mg is compiled using mg.exe, so that you can load information about photos into the Repository using a simple data format. This simple data format is a DSL (Domain Specific Language).

Next, the application starts the photo monitor. This means that it's now monitoring the content of the current directory (wherever PhotoStore.exe is) for new JPEG files.

---

Starting monitor...

Monitor started OK

 

Type <Ctrl>+C to stop monitor.

---

In a separate window, open PhotoStore\Photos directory. Drag and drop a JPEG file from there into PhotoStore\bin\Debug (or the directory where PhotoStore.exe is running).

Note: Remember to hold down the <Ctrl> key if using mouse, so you copy the file instead of moving it.

This is what the PhotoStore app should say:

---

Found new file <file>

Instance data saved to Gallery.input

 

Compiling photo data file with "M" grammar into "M" instance data file...

<command line for compilation>

 

"M" instance data file created OK

 

Compiling "M" instance data file into image...

<command line for compilation>

 

Image created OK

 

Loading instance data into the Repository...

<command line for loading>

 

Instance data loaded into the Repository OK

Yay!

 

Waiting for another photo...

Type <Ctrl>+C to stop monitor.

---

And this is what should have happened behind the scenes:

1.      When new file appears in current directory, the monitor detects its presence. Metadata from the photo are extracted and formatted into a simple input data file (Gallery.input). You can view this text filein any editor.

2.      Next, the input file is compiled by mgx.exe together with the grammar into an "M" instance data file Gallery.m.

Note: Because of slight discrepancies between "MGrammar" and the way "M" compiler works, the "M" instance data file has to be altered after it's created, so it can be compiled by the "M" compiler. See GalleryProper.m file for the final result, and the fix in code in PhotoLoader.cs, RunMToolChain().

3.      In the next step, GalleryProper.m is compiled using "M" compiler into Gallery.mx image, which is then loaded into the Repository using mx.exe. At this point, all metadata about the photo you just dropped into the current directory have been loaded into the Repository.

4.      In SQL Server Management Studio, navigate to the tables in Gallery schema and query them to see the photo metadata.

At this point we have created a schema in the Repository using "M" and mx.exe (Photo.m). We have also used a custom DSL (Gallery.input) and a custom grammar (Gallery.mg) to define information in a simple format and load it into the schema we just created using mx.exe. In the scenario from the PhotoStore.pptx, this is the point where the photographer has created a gallery for his own enjoyment.

Now let's take a look at what happens when the photographer wants to make his creations available to other users... for a price.

How to use Repository Security with PhotoStore

Repository security is used with PhotoStore when you want to limit access to photo information for users other than yourself. In order to utilize Repository security with the photos you just dropped into the current directory, do the following:

1.      Open CreateGalleryView.sql in SQL Server Management Studio. Make sure you are using Repository as the default database. Execute the query.

This will create a view on top of the existing tables in Gallery schema and grant SELECT permissions on this view to RepositoryReader. Now all users that are assigned the role of RepositoryReader will be able to read all public data in that view.

2.      In SQL Server Management Studio, navigate to [ConnectedDatabase] -> Security ->Logins and add another user with Windows credentials. Assign this user the role of RepositoryReader:

a.      Select User Mapping from the Select a Page list.

b.      Select the Users mapped to this login group box and choose Repository.

c.      Select the Database role membership for: Repository group box and choose RepositoryReader.

d.      Click OK.

Note:When you see this user's properties now, you will notice that under User Mapping -> Repository this user also has the role of RepositoryUser. This is correct, and it's part of the role hierarchy in the Repository.

3.      Log into Windows as the user you just added (let's call him Customer).

4.      Go to PhotoStore\WinGallery, open the solution and build it. Go to bin\Debug and run WinGallery.exe.

Notice that you (the Customer) can see photo metadata, but not the FullPath, where the photo is stored.

5.      Log into Windows as the previous user (let's call him Photographer).

6.      In SQL Server Management Studio, open GrantPermissions.sql file. Substitute DOMAIN\USER with the name of the Customer that you added above. Make sure you are using the Repository database and execute the query. The Customer now has permissions to view details of where the photograph is stored.

7.      Log into Windows as the Customer. Go to PhotoStore\WinGallery\bin\Debug and run WinGallery.exe.

Now you will be able to see the photograph's locations on disk!

8.      Log into Windows as the Photographer.

9.      In SQL Server Management Studio, open RevokePermissions.sql file. Substitute DOMAIN\USER with the name of the Customer that you added above, Make sure you are using the Repository database and execute the query. The Customer now has no permissions to view details of where the photograph is stored.

10.   Log into Windows as the Customer. Go to PhotoStore\WinGallery\bin\Debug and run WinGallery.exe.

You are again now not able to see the path to the photograph.

At this point we have demonstrated how, using the Repository security feature, we can limit access to certain data to different users. You can see the same result if you log in as Customer and try to view the raw data in the Repository using SQL Server Management Studio.

Have fun, or Custom Extensions

What if you could load the pictures themselves into the Repository, not just store the path?

The "M" compiler and mx.exe have the ability to load resources into the Repository, such as JPEG images. See the "M" compiler command line help for information on how to include resources in an image. When you compile an image with a resource, mx.exe will load that resource into Document.Documents table.

As an exercise and an extension to this app, you can try to incorporate resources into the PhotoStore application and make the photo itself available to the paying Customers.

If you take this challenge, let us know on the "Oslo" forum!

List of Files in PhotoStore Solution

CreateGalleryView.sql

This file contains T-SQL commands that create a view of Photo data in the Repository. The view combines abbreviated information about photos with full information, which can be filtered for different users by the means of the Repository security layer.

CreateSchema.bat

This file contains commands that compile Photo.m file into an image and install that image in the Repository, using the "M" compiler and mx. Installing the image creates the schema where the information about photos is stored.

DropGallerySchema.sql

These commands let you drop the Gallery schema so you can start again.

Gallery.mg

This is the grammar that defines the format in which photo information (instance data) will be loaded into the Repository. This allows us to use a very simple data format (all data in a row of strings) when adding information about new photos to the Repository.

GrantPermissions.sql

RevokePermissions.sql

These T-SQL commands grant and revoke permissions to read certain data in the Repository for a given user. Substitute [DOMAIN\USER] with your domain and user name, and run these selectively to either grant or revoke permissions.

Photo.m

This is the model of a Photo. It defines what information about a photograph is stored in the Repository, and in what way. It allows us to split information about a photograph into a set of abbreviated data (Photo), which contains selected photograph metadata, and full data (FullSizePhoto), which contains the physical path to the photo. It also contains information about the photographers.

Troubleshooting

Something Went Wrong, I Want to Start Again...

It's good to start with a clean slate. To drop the Gallery schema and all objects associated with it, open DropSchema.sql in SQL Server Management Studio and run it.

When I Drop a File into the Current Directory, PhotoStore.exe says:

---

Compiling input file failed. Skipping.

---

This means that the input file did not compile correctly in mgx. This might be an incompatibility between the mgx.exe version you have installed and the version for which PhotoStore was written.

Go to the "Oslo" forum and ask about this problem.
